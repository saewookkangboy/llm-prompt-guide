04

  

    
STRUCTURE / 설계

    
효과적인 프롬프트 구조 설계

    
좋은 프롬프트는 4개의 덩어리로 구성됩니다 — 길어질수록 이 구조가 출력 안정성에 결정적입니다

    

      

        

          
01

          

            
ROLE

            
역할 설정 (System Prompt)
"너는 B2B SaaS 10년 경력 CMO야, 데이터 기반 의사결정을 중시해"처럼 구체적으로. "마케터"는 X.

          

        

        

          
02

          

            
CONTEXT

            
맥락 & 배경 정보
현재 상황, 대상 독자, 기존 작업물, 제약 조건. LLM은 당신의 상황을 모릅니다. Claude는 풍부한 맥락일수록 좋은 결과.

          

        

        

          
03

          

            
TASK

            
작업 지시 (명확 + 단수)
복합 작업은 단계별 분리. "분석하고, 요약하고, 추천해줘" → 3개의 별도 프롬프트로 나누면 품질 ↑.

          

        

        

          
04

          

            
FORMAT

            
출력 형식 명시
JSON 형식으로, 마크다운 표로, 3문장 이내로 — 원하는 형식을 정확히 지정.

          

        

        

          
05

          

            
CONSTRAINTS

            
제약 & 금지사항
"추측하지 마세요", "서론 없이 바로 결과만". 네거티브 프롬프트는 출력 품질 안정화에 효과적.

          

        

      

      

        
Pro Tip · Few-shot
예시 추가로 품질 ↑↑
원하는 출력 형식의 예시 1~3개를 프롬프트에 포함. 설명 100줄보다 예시 1개가 효과적.

        
Pro Tip · 존댓말
해요체로 작성하세요
완결된 문장은 모델이 범위와 의도를 정확히 파악. 개조식 나열보다 해요체 요청문이 일관된 결과.

        
Anti-Pattern
모호한 지시 금지
"좋은 글 써줘" → ❌
좋다의 기준·길이·톤·독자를 모두 명시해야.

WORKFLOW / 자동화

    
LLM 워크플로 자동화 패턴

    

      

        
01

        

          
체인 프롬프팅 (Chain-of-Thought)

          
복잡한 작업을 분석 → 계획 → 실행 → 검토 단계로 분리. 각 단계의 출력이 다음 단계의 입력. 중간 결과를 저장하면 디버깅이 용이합니다.

        

        
CoT

      

      

        
02

        

          
RAG (Retrieval-Augmented Generation)

          
LLM에 없는 최신·내부 정보를 벡터 DB 검색 → 컨텍스트 주입 → 생성으로 보완. Hallucination을 줄이고 도메인 특화 응답 품질을 높입니다.

        

        
RAG

      

      

        
03

        

          
멀티 에이전트 오케스트레이션

          
역할별 전문 에이전트 구성: 조사 → 분석 → 작성 → 검토 에이전트. n8n, LangGraph, Harness Engineering 등으로 구현.

        

        
Agent

      

      

        
04

        

          
Self-Consistency & 앙상블

          
동일 프롬프트를 여러 번 실행 → 다수결 또는 평가 LLM이 최적 답 선택. 정확성이 중요한 작업에 적용.

        

        
Ensemble

PRACTICE / 실무

    
실무 프롬프팅 Do & Don't

    

      

        
✅ DO — 이렇게 하세요

        
✅
구체적인 역할과 전문성 수준을 명시하세요.
"마케터"보다 "10년 경력 B2B SaaS 마케터, NPS 개선 전문가"처럼.

        
✅
출력 형식을 정확히 지정하세요.
JSON, 마크다운 표, 불릿 5개, 300자 이내 등 구체적 형식 명시.

        
✅
Few-shot 예시를 포함하세요.
원하는/원하지 않는 출력 예시를 함께 제공하면 품질이 크게 향상.

        
✅
모호한 용어를 정의하세요.
"좋은", "전문적인"처럼 기준 없는 수식어는 구체적 기준으로 대체.

      

      

        
❌ DON'T — 이건 피하세요

        
❌
한 번에 너무 많은 작업을 요청하지 마세요.
"분석하고, 요약하고, 번역하고, 이미지도 만들어줘" → 단계별 분리.

        
❌
막연한 품질 요구를 하지 마세요.
"최선을 다해줘" — 모델은 기준이 없으면 자체 판단으로 처리.

        
❌
모든 모델에 같은 프롬프트를 사용하지 마세요.
Claude와 GPT-4o는 선호하는 입력 스타일이 다릅니다. 모델별 최적화 필요.

        
❌
첫 번째 결과를 그대로 사용하지 마세요.
항상 검증 루프를 거쳐 최소 2~3회 반복 개선 후 최종 활용.

      

    

  







  
09

  

    
PRINCIPLES / 요약

    
핵심 원칙 요약

    

      
원칙 01
모델 이해가 먼저
  • 작업 유형에 맞는 모델 선택이 프롬프트 품질보다 중요할 수 있음
  • 비용·지연시간을 고려한 라우팅 전략 설계 필수

      
원칙 02
구조화된 프롬프트
  • 역할→맥락→태스크→형식→제약의 5단계 구조 준수
  • Few-shot 예시 포함으로 출력 품질 안정화

      
원칙 03
검증 루프 체계화
  • 첫 출력을 최종 결과로 사용하지 않는 원칙
  • 통과된 프롬프트는 템플릿화하여 팀 자산 축적

      
원칙 04
워크플로 자동화
  • 반복 작업은 체인 프롬프팅으로 자동화
  • 멀티 에이전트로 복잡한 업무 파이프라인 구축

      
원칙 05
지속적 개선
  • 프롬프트를 버전 관리하고 성능 지표 추적
  • 모델 업데이트 시 기존 프롬프트 재검증 필수

      
원칙 06
책임감 있는 활용
  • 중요 의사결정에 LLM 단독 사용 금지
  • AI 생성 콘텐츠임을 투명하게 공시

PRACTICE 01 / 일상생활

    
일상생활 프롬프트 실습

    
ChatGPT 5.5 · Claude · Gemini 공용 — 역할·데이터·제약·출력 4덩어리 구조를 일상에 적용

    

      

        
예시 ① — 여행 계획

        
전주 2박 3일 혼자 여행

        

          
PROMPT🔍 클릭해서 확대

          
ROLE 당신은 국내 여행 전문 플래너예요.

저는 혼자 2박 3일로 전주를 여행하려고 해요.
이동은 대중교통만 사용하고, 하루 예산은 8만 원이에요.

다음 조건을 지켜 주세요:
- 유명 관광지보다 현지인이 즐기는 곳을 우선해 주세요.
- 식사는 끼니당 1만 5천 원 이하로 추천해 주세요.

결과는 다음 형식으로 작성해 주세요:
1. 일자별 동선 요약
2. 추천 식당 및 카페 (이유 포함)
3. 예산 요약표

불확실한 영업 정보는 "확인 필요"로 표시해 주세요.

        

        

          Claude: 개인취향 상세 추가  |  Gemini: Maps 연동

      최종 점검: "이 프롬프트를 처음 보는 사람이 아무 질문 없이 내가 원하는 결과를 만들어낼 수 있는가?" — YES라면 발송, NO라면 빠진 맥락을 보완하세요.
    

  







  
해요

  

    
LANGUAGE / 해요체

    
프롬프트를 해요체로 써야 하는 이유

    
명령조·개조식 나열보다 완결된 요청문이 일관되고 예측 가능한 출력을 만들어 냅니다


    

      


        

          
이유 ① — 훈련 데이터 정렬

          
고품질 텍스트 패턴 활성화

          
LLM은 인터넷의 자연어 텍스트로 훈련됩니다. 대부분의 고품질 한국어 텍스트(블로그·문서·설명글)는 해요체로 작성되어 있어요. 해요체 입력은 모델이 고품질 응답 패턴을 활성화하는 데 더 잘 정렬됩니다.

        


        

          
이유 ② — 맥락의 명확성

          
문장 구조가 완결되어 의도 전달 ↑

          
해요체는 문장 구조가 완결되어 있어 모델이 요청의 범위와 의도를 더 정확하게 파악해요. 반면 개조식 단어 나열 "요약, 3줄, 간결"은 해석의 여지가 많아 출력이 흔들립니다.

        


        

          
이유 ③ — 협력적 상호작용 유도

          
지시 → 협력 관계로 전환

          
해요체는 모델과의 상호작용을 지시가 아닌 협력 관계로 설정해요. 이는 모델이 단순 실행이 아니라 의도를 이해하고 판단하는 방식으로 응답하도록 유도합니다.

        


        

          
이유 ④ — 작성자의 사고 정리

          
프롬프트 쓰면서 의도가 명확해짐

          
해요체로 완성된 문장을 쓰다 보면 작성자 스스로 "내가 무엇을 원하는가"를 더 명확히 정리하게 돼요. 이 자체가 컨텍스트 품질 향상으로 이어집니다.

        


      


      


        
        

          

            반말 vs 해요체 — 출력 결과 비교
          


          
          

            
❌ 반말 (명령조)

            

              마케팅 전략 분석해.

              경쟁사 비교, 3가지, 표로.
            

            

              
입력 토큰 ~12

              
출력 품질 모호·불안정

              
재질문 확률 높음

            

          


          
          

            
✅ 해요체 (요청문)

            

              당신은 B2B 마케팅 전문가예요.

              저희 제품의 마케팅 전략을 분석해 주세요.

              경쟁사 3곳과 비교하고, 표 형식으로

              강점·약점·기회를 정리해 주세요.
            

            

              
입력 토큰 ~38

              
출력 품질 명확·안정

              
재질문 확률 낮음

            

          

        


        
        

          
토큰 투자 인사이트

          

            

              
+26

              
추가 입력 토큰

            

            

              
÷ 3

              
재질문 횟수 감소

            

            

              
↓ 60%

              
총 토큰 절감

            

          

          

            결론: 입력 토큰 +26개로 재질문 2~3회를 없애면, 총 사용 토큰이 오히려 절감돼요. 해요체는 비용 효율적인 선택입니다.
          

        


      

    

  







  
턴

  

    
TURN / 대화 구조

    
멀티턴 vs 싱글턴 — 언제 무엇을 쓸까요?

    
대화 구조 선택이 출력 품질과 비용을 결정합니다


    

      

        
싱글턴 (Single-turn)1회 완결

        
하나의 프롬프트 → 하나의 완전한 응답

        
    
          
  • 모든 맥락을 한 번에 담아 전송해요
    • 
          
  • 독립적·반복 가능한 작업에 적합해요
    • 
          
  • API 비용 예측이 쉬워요
    • 
          
  • 응답이 길고 구조화되어 있어요
    • 
          
  • 이전 대화 없이 매번 처음부터 시작해요
    • 
        

        

          최적 상황

          문서 작성, 코드 생성, 번역, 요약,
단발성 분석, 자동화 파이프라인
        

      


      

        
멀티턴 (Multi-turn)대화 연속

        
여러 번의 대화를 통해 점진적으로 발전

        
    
          
  • 이전 응답을 컨텍스트로 누적해요
    • 
          
  • 복잡한 요구사항을 단계적으로 정제해요
    • 
          
  • 모델이 사용자 스타일을 학습해요
    • 
          
  • 피드백 루프로 품질을 반복 개선해요
    • 
          
  • 긴 프로젝트·창작 작업에 강해요
    • 
        

        

          최적 상황

          전략 기획, 창작 개선, 디버깅,
복잡한 의사결정, 교육·학습 세션
        

      

    


    
    

      

        
항목싱글턴멀티턴

        
          

            컨텍스트 관리
            프롬프트에 모두 포함
            대화 히스토리 자동 누적
          

          

            토큰 비용
            예측 가능·단발 비용
            턴마다 누적 → 비용 증가
          

          

            오류 수정
            프롬프트 재작성 필요
            후속 메시지로 즉시 수정
          

          

            자동화 적합성
            파이프라인·배치 처리 최적
            사람이 개입하는 워크플로에 적합
          

        
      

    

  







  
전략

  

    
STRATEGY / 실전

    
싱글턴 · 멀티턴 최적 전략

    
목적에 맞는 방식을 선택하고, 각 방식의 패턴을 적용하면 품질이 극적으로 향상돼요


    


      
      

        
싱글턴 — 한 번에 완성하는 전략


        

          
패턴 ① — 완전한 컨텍스트 선언

          
역할·배경·목적·독자·제약·형식을 한 번의 프롬프트에 모두 담아 주세요. 빈칸이 있으면 모델이 추측하고, 추측은 품질 저하로 이어져요.

          

            
SINGLE-TURN🔍 클릭해서 확대

            
ROLE 당신은 B2B SaaS 기업의 CMO예요.
배경 우리 제품의 6월 MAU는 12,000명,
     전월 대비 8% 감소했어요.
목적 경영진 보고용 원인 분석 및 개선안
제약 500자 이내, 추측은 "추정" 표기,
     경쟁사 이름 언급 금지
형식 ① 원인 분석 ② 개선안 3가지
     ③ 우선순위표 (표 형식)

          

        


        

          
패턴 ② — 계획 선제시 + 승인 후 실행

          
싱글턴이지만 2단계로 분리해요. 먼저 계획만 요청 → 승인 후 본문 작성. 재작업을 줄이는 가장 효과적인 싱글턴 패턴이에요.

          

            
SINGLE-TURN · 2단계🔍 클릭해서 확대

            
당신은 [역할]이에요.
[배경과 목적을 설명해 주세요]

먼저 아래 항목만 제시해 주세요:
- 작업 이해 요약 (1문장)
- 진행 계획 (목차/단계)
- 예상 결과물 형식

제가 "OK"라고 하면 본문을 작성해 주세요.

          

        

      


      
      

        
멀티턴 — 단계적으로 발전시키는 전략


        

          
패턴 ③ — 역할 고정 후 대화 시작

          
멀티턴의 첫 메시지에서 역할·규칙·출력 형식을 고정해 주세요. 이후 대화에서는 데이터와 요청만 추가하면 돼요. 첫 턴이 전체 품질을 결정해요.

          

            
MULTI-TURN · 1턴🔍 클릭해서 확대

            
오늘 세션의 규칙을 먼저 설정할게요.

당신은 [역할]이에요.
이 대화에서 다음 규칙을 항상 지켜 주세요:
- 모든 답변은 해요체로 작성해 주세요.
- 불확실한 내용은 "추정"으로 표기해 주세요.
- 출력은 항상 [형식]으로 작성해 주세요.

규칙을 이해했으면 "시작할게요"라고 답해 주세요.

          

        


        

          
패턴 ④ — 점진적 정제 (Iterative Refinement)

          
초안 → 피드백 → 개선을 반복해요. 각 턴에서 "이전 답변을 기반으로"를 명시하면 컨텍스트 손실을 방지할 수 있어요.

          

            
MULTI-TURN · 2턴+🔍 클릭해서 확대

            
이전 답변을 기반으로 아래를 수정해 주세요.

수정 사항:
- [구체적인 수정 요청]
- 톤을 더 [방향]으로 바꿔 주세요.

유지할 사항:
- [변경하지 않을 부분]은 그대로 두어 주세요.

          

        


        
        

          
언제 무엇을 선택할까요?

          

            

              →
              자동화·반복 작업이라면 싱글턴 — 파이프라인에 넣기 쉬워요
            

            

              →
              복잡하고 모호한 요구사항이라면 멀티턴 — 단계적 정제가 더 안전해요
            

            

              →
              긴 창작·전략 기획이라면 멀티턴 — 대화를 통한 아이디어 발전이 강해요
            

            

              →
              토큰 비용이 중요하다면 싱글턴 + 계획 선제시 패턴이 가장 경제적이에요
