F5

  

    
FABLE 5 / 프롬프트 설계

    
Fable 5에서만 되는 4가지 패턴

    
Claude Opus 4.8 / Sonnet 4.6과 다르게 Fable 5만 가진 100만 토큰·자기 검증·복합 자율 추론을 이끌어 내는 전용 프롬프트 설계법


    

      

        
1

        

          
초장문

          

            
컨텍스트 압축 요청 — 100만 토큰 활용법

            
문서 전체를 첨부한 뒤 먼저 전체 구조를 파악하고, 그 다음 아래 세 가지를 추출해 주세요처럼 2단계로 지시해요. 토큰 한계를 걱정하지 말고 원문을 자르지 마세요.

          

        

      

      

        
2

        

          
자기검증

          

            
검증 루프 내장 — 스스로 오류를 찾게 하기

            
답변 마지막에 이 답변에서 논리 오류나 반례가 있다면 스스로 지적해 주세요를 붙이면 Fable 5는 자체 검증 단계를 실행해요. 다른 모델보다 정확도가 크게 높아집니다.

          

        

      

      

        
3

        

          
복합추론

          

            
조건 충돌 해결 — 복잡한 요구사항 자율 정리

            
서로 충돌하는 조건이 있을 때 제약 간 우선순위를 스스로 정하고 결정 이유를 설명해 주세요라고 넣으면 Fable 5가 자율적으로 해결안을 도출해요.

          

        

      

      

        
4

        

          
멀티라운드

          

            
장기 프로젝트 선언 — 대화 전체를 하나의 작업으로

            
첫 턴에 이 대화는 X 프로젝트 전체를 다룹니다. 이전 응답을 항상 참조하며 일관성을 유지해 주세요라고 선언하면 Fable 5가 대화 전체를 하나의 작업으로 취급해요.

          

        

      

    


    

      언제 Fable 5를 선택해야 하나요? 문서가 50페이지 이상이거나, 조건이 5개 이상 충돌하거나, 한 대화에서 여러 작업을 이어서 해야 할 때예요. 그 외엔 Opus 4.8 / Sonnet 4.6이 더 빠르고 경제적이에요.
    

  








  
04

  

    
STRUCTURE / 설계

    
효과적인 프롬프트 구조 설계

    
좋은 프롬프트는 4개의 덩어리로 구성됩니다 — 길어질수록 이 구조가 출력 안정성에 결정적입니다

    

      

        

          
01

          

            
ROLE

            
역할 설정 (System Prompt)
"너는 B2B SaaS 10년 경력 CMO야, 데이터 기반 의사결정을 중시해"처럼 구체적으로. "마케터"는 X.

          

        

        

          
02

          

            
CONTEXT

            
맥락 & 배경 정보
현재 상황, 대상 독자, 기존 작업물, 제약 조건. LLM은 당신의 상황을 모릅니다. Claude는 풍부한 맥락일수록 좋은 결과.

          

        

        

          
03

          

            
TASK

            
작업 지시 (명확 + 단수)
복합 작업은 단계별 분리. "분석하고, 요약하고, 추천해줘" → 3개의 별도 프롬프트로 나누면 품질 ↑.

          

        

        

          
04

          

            
FORMAT

            
출력 형식 명시
JSON 형식으로, 마크다운 표로, 3문장 이내로 — 원하는 형식을 정확히 지정.

          

        

        

          
05

          

            
CONSTRAINTS

            
제약 & 금지사항
"추측하지 마세요", "서론 없이 바로 결과만". 네거티브 프롬프트는 출력 품질 안정화에 효과적.

          

        

      

      

        
Pro Tip · Few-shot
예시 추가로 품질 ↑↑
원하는 출력 형식의 예시 1~3개를 프롬프트에 포함. 설명 100줄보다 예시 1개가 효과적.

        
Pro Tip · 존댓말
해요체로 작성하세요
완결된 문장은 모델이 범위와 의도를 정확히 파악. 개조식 나열보다 해요체 요청문이 일관된 결과.

        
Anti-Pattern
모호한 지시 금지
"좋은 글 써줘" → ❌
좋다의 기준·길이·톤·독자를 모두 명시해야.

      

    

  







  
해요

  

    
LANGUAGE / 해요체

    
프롬프트를 해요체로 써야 하는 이유

    
명령조·개조식 나열보다 완결된 요청문이 일관되고 예측 가능한 출력을 만들어 냅니다


    

      


        

          
이유 ① — 훈련 데이터 정렬

          
고품질 텍스트 패턴 활성화

          
LLM은 인터넷의 자연어 텍스트로 훈련됩니다. 대부분의 고품질 한국어 텍스트(블로그·문서·설명글)는 해요체로 작성되어 있어요. 해요체 입력은 모델이 고품질 응답 패턴을 활성화하는 데 더 잘 정렬됩니다.

        


        

          
이유 ② — 맥락의 명확성

          
문장 구조가 완결되어 의도 전달 ↑

          
해요체는 문장 구조가 완결되어 있어 모델이 요청의 범위와 의도를 더 정확하게 파악해요. 반면 개조식 단어 나열 "요약, 3줄, 간결"은 해석의 여지가 많아 출력이 흔들립니다.

        


        

          
이유 ③ — 협력적 상호작용 유도

          
지시 → 협력 관계로 전환

          
해요체는 모델과의 상호작용을 지시가 아닌 협력 관계로 설정해요. 이는 모델이 단순 실행이 아니라 의도를 이해하고 판단하는 방식으로 응답하도록 유도합니다.

        


        

          
이유 ④ — 작성자의 사고 정리

          
프롬프트 쓰면서 의도가 명확해짐

          
해요체로 완성된 문장을 쓰다 보면 작성자 스스로 "내가 무엇을 원하는가"를 더 명확히 정리하게 돼요. 이 자체가 컨텍스트 품질 향상으로 이어집니다.

        


      


      


        
        

          

            반말 vs 해요체 — 출력 결과 비교

              경쟁사 3곳과 비교하고, 표 형식으로

STRATEGY / 실전

    
싱글턴 · 멀티턴 최적 전략

    
목적에 맞는 방식을 선택하고, 각 방식의 패턴을 적용하면 품질이 극적으로 향상돼요


    


      
      

        
싱글턴 — 한 번에 완성하는 전략


        

          
패턴 ① — 완전한 컨텍스트 선언

          
역할·배경·목적·독자·제약·형식을 한 번의 프롬프트에 모두 담아 주세요. 빈칸이 있으면 모델이 추측하고, 추측은 품질 저하로 이어져요.

          

            
SINGLE-TURN🔍 클릭해서 확대

            
ROLE 당신은 B2B SaaS 기업의 CMO예요.
배경 우리 제품의 6월 MAU는 12,000명,
     전월 대비 8% 감소했어요.
목적 경영진 보고용 원인 분석 및 개선안
제약 500자 이내, 추측은 "추정" 표기,
     경쟁사 이름 언급 금지
형식 ① 원인 분석 ② 개선안 3가지
     ③ 우선순위표 (표 형식)

          

        


        

          
패턴 ② — 계획 선제시 + 승인 후 실행

          
싱글턴이지만 2단계로 분리해요. 먼저 계획만 요청 → 승인 후 본문 작성. 재작업을 줄이는 가장 효과적인 싱글턴 패턴이에요.

          

            
SINGLE-TURN · 2단계🔍 클릭해서 확대

            
당신은 [역할]이에요.
[배경과 목적을 설명해 주세요]

먼저 아래 항목만 제시해 주세요:
- 작업 이해 요약 (1문장)
- 진행 계획 (목차/단계)
- 예상 결과물 형식

제가 "OK"라고 하면 본문을 작성해 주세요.

          

        

      


      
      

        
멀티턴 — 단계적으로 발전시키는 전략


        

          
패턴 ③ — 역할 고정 후 대화 시작

          
멀티턴의 첫 메시지에서 역할·규칙·출력 형식을 고정해 주세요. 이후 대화에서는 데이터와 요청만 추가하면 돼요. 첫 턴이 전체 품질을 결정해요.

          

            
MULTI-TURN · 1턴🔍 클릭해서 확대

            
오늘 세션의 규칙을 먼저 설정할게요.

당신은 [역할]이에요.
이 대화에서 다음 규칙을 항상 지켜 주세요:
- 모든 답변은 해요체로 작성해 주세요.
- 불확실한 내용은 "추정"으로 표기해 주세요.
- 출력은 항상 [형식]으로 작성해 주세요.

규칙을 이해했으면 "시작할게요"라고 답해 주세요.

          

        


        

          
패턴 ④ — 점진적 정제 (Iterative Refinement)

          
초안 → 피드백 → 개선을 반복해요. 각 턴에서 "이전 답변을 기반으로"를 명시하면 컨텍스트 손실을 방지할 수 있어요.

          

            
MULTI-TURN · 2턴+🔍 클릭해서 확대

            
이전 답변을 기반으로 아래를 수정해 주세요.

수정 사항:
- [구체적인 수정 요청]
- 톤을 더 [방향]으로 바꿔 주세요.

유지할 사항:
- [변경하지 않을 부분]은 그대로 두어 주세요.

PRACTICE / 실무

    
실무 프롬프팅 Do & Don't

    

      

        
✅ DO — 이렇게 하세요

        
✅
구체적인 역할과 전문성 수준을 명시하세요.
"마케터"보다 "10년 경력 B2B SaaS 마케터, NPS 개선 전문가"처럼.

        
✅
출력 형식을 정확히 지정하세요.
JSON, 마크다운 표, 불릿 5개, 300자 이내 등 구체적 형식 명시.

        
✅
Few-shot 예시를 포함하세요.
원하는/원하지 않는 출력 예시를 함께 제공하면 품질이 크게 향상.

        
✅
모호한 용어를 정의하세요.
"좋은", "전문적인"처럼 기준 없는 수식어는 구체적 기준으로 대체.

      

      

        
❌ DON'T — 이건 피하세요

        
❌
한 번에 너무 많은 작업을 요청하지 마세요.
"분석하고, 요약하고, 번역하고, 이미지도 만들어줘" → 단계별 분리.

        
❌
막연한 품질 요구를 하지 마세요.
"최선을 다해줘" — 모델은 기준이 없으면 자체 판단으로 처리.

        
❌
모든 모델에 같은 프롬프트를 사용하지 마세요.
Claude와 ChatGPT 5.5는 선호하는 입력 스타일이 다릅니다. 모델별 최적화 필요.

        
❌
첫 번째 결과를 그대로 사용하지 마세요.
항상 검증 루프를 거쳐 최소 2~3회 반복 개선 후 최종 활용.

LOOP ENGINEERING / 개요

    
에이전트를 직접 프롬프트하는 시대는 끝났다

    
Loop Engineering — 당신이 에이전트에게 프롬프트를 넣는 사람을 시스템으로 대체하는 설계 철학


    

      

        
정의

        

          Loop Engineering은 반복적인 목표를 정의하고, AI가 하위 에이전트·검증·외부 상태를 활용하여 목표를 달성하거나 인간에게 전달할 때까지 반복 실행하는 자율 제어 시스템을 설계하는 것입니다.
        

      

      

        

          
PETER STEINBERGER

          
"You shouldn't be prompting coding agents anymore. You should be designing loops that prompt your agents."

        

        

          
BORIS CHERNY — HEAD OF CLAUDE CODE, ANTHROPIC

LOOP ANATOMY / 구조 다이어그램

    
Loop의 해부학

    
단일 Loop가 목표를 달성하거나 인간에게 전달하기까지의 전체 흐름


    
    

      
    


    

      

        
PHASE 1

        
발견 → 상태 기록

        
스케줄에 의해 시작, 트리아지 스킬로 작업 파악, STATE.md 업데이트

      

      

        
PHASE 2

        
구현 → 검증

        
격리된 워크트리에서 구현 에이전트 실행, 검증 에이전트가 테스트·게이트 통과 확인

      

      

        
PHASE 3

        
게이트 → 반복

        
안전하면 자동 커밋·PR, 위험하면 전체 컨텍스트와 함께 인간에게 에스컬레이션 후 재시작

영속적 상태 파일

          

        

        
모델은 세션 간 기억이 없습니다. 루프는 STATE.md나 DB에 현재 작업·결과·대기 항목을 기록해야 합니다. 이 파일이 루프의 척추입니다.

        

          STATE.md
          LOOP-STATE.json
        

      

    


    

      최소 viable loop: 스케줄링 + 스킬(트리아지) + STATE 파일부터 시작 → 워크트리 → 서브에이전트 검증 → MCP 순서로 점진 도입

PATTERNS / 생산 루프 패턴

    
7가지 생산 루프 패턴

    
L1(리포트 전용) → L2(보조 수정) → L3(무인 실행) 단계적 확장 전략


    

      

        
          

            패턴
            주기
            첫 주 목표
            토큰 비용
            설명
          

        
        
          

            Daily Triage L1
            1d – 2h
            리포트만
            낮음
            이슈·PR을 트리아지하고 STATE.md 업데이트. 인간이 주간 리뷰 후 결정. 첫 루프로 추천.
          

          

            PR Babysitter L2
            5 – 15m
            L1 모니터링
            높음
            활성 PR 상태를 주기적으로 확인. 워크트리에서 수정 제안, 검증 필수, 자동 머지 없음.
          

          

            CI Sweeper L2
            5 – 15m
            L2 신중하게
            매우 높음
            CI 실패를 자동 감지·수정 시도. 검증 에이전트 필수. 토큰 비용이 크므로 활성화 타이밍 주의.
          

          

            Dependency Sweeper L2
            6h – 1d
            패치만
            중간
            패치 버전·저위험 CVE만 자동 업데이트. 메이저 버전·블랙리스트 패키지는 인간 게이트.
          

          

            Changelog Drafter L1
            1d / 릴리즈
            초안만
            낮음
            릴리즈 노트 초안 자동 생성. 인간이 승인 후 게시. Daily Triage와 조합하면 최고 가성비.
          

          

            Post-Merge Cleanup L1
            1d – 6h
            오프피크
            낮음
            머지 후 불필요한 브랜치·임시 코드 정리. 오프피크 시간대에 실행해 충돌 최소화.
          

          

            Issue Triage L1
            2h – 1d
            제안만
            낮음
            새 이슈 자동 분류·라벨링·우선순위 제안. Daily Triage의 특화 버전으로 함께 운영 가능.
          

        
      

    


    

      

        
L1 — REPORT ONLY

        
루프가 관찰·리포트만 합니다. 인간이 모든 액션을 결정. 신뢰를 쌓는 시작 단계.

      

      

        
L2 — ASSISTED FIXES

        
루프가 수정을 제안하거나 적용하되 검증 필수. 민감한 변경은 인간 게이트. 신뢰 검증 단계.

      

      

        
L3 — UNATTENDED

        
허용목록 내 액션을 무인 실행. 강력한 검증 에이전트와 완벽한 denylist가 전제. 성숙 단계.
